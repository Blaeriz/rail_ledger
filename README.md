# Rail Ledger

## Contribution Guidelines

Everyone is free to work on **any feature, bug, or improvement** they’d like 🎉.  
To keep things smooth and consistent, please follow these rules:

1. **Open an Issue First**  
   - Always create a GitHub Issue before starting work.  
   - Keeps the project organized and avoids duplicate efforts.

2. **Commits**  
   - Use [Commitizen](https://commitizen-tools.github.io/commitizen/) for all commits.  
   - Keep commits **small and focused** (bite-sized).  
   - Mention if code was **AI-assisted** in the commit message (e.g., `feat(api): add login [AI-assisted]`).  

3. **Code Formatting & Style**  
   - Run **Prettier** before committing.  
   - No inconsistent formatting in PRs.  

4. **Pull Requests (PRs)**  
   - Link PRs to the related Issue.  
   - Keep PRs **small and reviewable**.  
   - No WIP PRs without a clear plan.

5. **General Guidelines**  
   - **Write meaningful commit messages** → “fix bug” is not meaningful.  
   - **Don’t commit directly to `main`** → always use a branch + PR.  
   - **Add comments when code isn’t obvious** → future-you will thank you.  
   - **Tests where it makes sense** → if you add logic, add at least basic tests.  
   - **No secrets in the repo** → don’t commit `.env` or API keys.  
   - **Respect the team’s time** → keep discussions/issues focused and productive.  

4. **Branch Naming Convention**  
   - Use clear and consistent branch names.  
   - Format: type/short-description  
   - Examples:
               feat/add-login-api
               fix/railpad-scan-bug
               chore/update-readme
   - Match type with commit types (feat, fix, chore, docs, refactor, test).
---

✅ Stick to these guidelines → the project stays clean, collaborative, and easy to maintain.  





Thanks for contributing 🚀
